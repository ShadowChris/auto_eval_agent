# auto_eval_agent · 操作类（录屏）评测

> 操作类 query（设闹钟、下单、改设置、导航…）往往拿不到结构化日志/结果、也难以沙箱重放，唯一可靠的观测是**录屏**。本模式（`mode=operation`）通过「ffmpeg 场景抽帧 → image_url 多图喂裁判 → 判断操作是否完成」把录屏评测接入框架的多裁判盲评/聚合/元评测体系。

## 一、整体流程

```
前端逐题卡片，或 JSONL 清单（query + context + video_path + agent_statement）
   │  单条：POST /api/upload/video
   │  批量导入：POST /api/parse（只解析，不抽帧）
   ▼
submit: POST /api/eval {mode:"operation", items:[{query, context?, video_path, answer?}]}
   ▼
按 Web 会话逐题校验视频 → 场景抽帧 → 写回历史快照             web/runner.py
   │  runs/videos/imported/<session_name>/<序号>_<item_id>/
   ▼
_eval_one(operation)
   │  _to_evalitem → EvalItem(media, metadata.frames)
   │  对每个裁判并发：RubricJudge.score(item, answer, eval_mode="operation")
   ▼
score 内：抽帧路径 → encode_frame(base64) → complete(user_images=...)   rubric_judge.py
   │  裁判看多图 + OPERATION prompt → 输出 {rubric, total, correctness, rationale}
   ▼
aggregate_scores(多裁判) → 可选仲裁 → SSE 推结果                      ensemble.py
```

涉及的核心文件：

| 文件 | 职责 |
|---|---|
| `src/auto_eval/media.py` | 场景抽帧 + 帧编码（纯 ffmpeg + PIL） |
| `src/auto_eval/judges/base.py` | `JudgeClient.complete`（多模态 + agent loop） |
| `src/auto_eval/judges/prompts.py` | `OPERATION_SYSTEM` / `OPERATION_USER` |
| `src/auto_eval/judges/rubric_judge.py` | `score` 的 `eval_mode="operation"` 分支 |
| `config/skills/operation.yaml` | 操作类评分维度（可配置） |
| `src/auto_eval/web/server.py` | 单条上传、解析与评估接口 |
| `src/auto_eval/web/operation_media.py` | 本地视频路径校验与按会话抽帧 |
| `src/auto_eval/web/runner.py` | 开始评估时准备批量视频、执行 operation 评测 |

## 二、关键帧抽取（media.py）

> 裁判看录屏 = 把视频转成一组图片喂它。抽哪些帧、抽几帧，直接决定评测的**准**（有没有抓住操作关键瞬间）和**省**（token 成本）。本节详述「场景检测 + 去重 + 上限/保底 + 终态保底」方案：为什么这么设计、和"一般的抽帧"比好在哪里。

### 2.1 要解决的问题：为什么不能等间隔抽帧

操作录屏的画面信息分布有一个鲜明特征——**「长静止 + 突发跳变」**：用户在一个界面停留思考（画面纹丝不动，可能持续数秒），偶尔快速点击、跳转、弹窗（画面瞬间剧变，往往 < 0.5 秒）。评测真正关心的"操作有没有发生、做到哪一步、最终态对不对"，**几乎全部信息都集中在那些短暂的跳变瞬间**；静止段是纯噪声。

而"一般抽帧"（等时间间隔均匀采样）的隐含假设是**「信息在时间上均匀分布」**——这恰好和操作录屏的真实分布**反过来**。于是等间隔抽帧会犯两个错：

| 错误 | 表现 |
|---|---|
| 静止段浪费 | 在数秒不变的界面里连抽几帧，几乎完全重复，白烧 token |
| 跳变段漏帧 | 两次采样之间发生的关键操作（如弹窗一闪、一次点击）被跳过 |

**结论：抽帧策略必须「跟着内容走」，而不是「跟着时间走」。** 这是采用「场景变化检测」的根本理由。

### 2.2 核心原理：场景变化检测（scene change detection）

**ffmpeg 的 `scene` 分数是什么。** `select` 滤镜给每一帧算一个 `scene` 分数（0–1），度量**这一帧相对上一帧的画面变化程度**——内部基于帧间像素差异（SAD，绝对差值之和）做归一化：画面几乎不变（静止段）→ 接近 0；画面显著变化（界面跳转、弹窗、滚动）→ 明显升高（0.1–0.3+）。所以「scene 高的帧」≈「画面跳变的帧」≈「操作录屏里有信息量的帧」，是**内容驱动的信号**。

**两条滤镜：`select` + `showinfo`：**

```
ffmpeg -i video.mp4 -an -vf "select='gt(scene,0.10)',showinfo" -f null -
```

- `select='gt(scene,0.10)'`：只放行 scene 分数 > 0.10 的帧（画面变化超过阈值的时刻）；
- `showinfo`：把每一个被放行帧的信息（含 `pts_time` 秒数）打到 stderr；
- `-f null -`：不真正编码输出，只跑滤镜拿 stderr（轻量、快）。

用正则 `pts_time:([0-9.]+)` 解析 stderr，得到**所有"画面显著变化"的时刻列表**——这就是关键帧候选。

> Windows 坑：`subprocess` 默认用 GBK 解码 ffmpeg stderr，遇到非 GBK 字节会失败使 `stderr=None`。`media.py` 显式 `encoding="utf-8", errors="replace"`。

**为什么 scene 检测天然适合操作录屏：** 长静止段 scene≈0 不抽（省 token），跳变段 scene 高抽（抓住关键瞬间）——自动把采样预算花在有信息的地方，等间隔做不到这种区分。

### 2.3 实现详解：`select_keyframe_times` 的五步流程

候选时刻拿到后还要四步加工。完整流程：

```
scene_change_times → _dedup → 上限抽样 → 保底补帧 → 终态保底
   (拿跳变时刻)      (合并相邻) (太多则均匀减) (太少则均匀补) (强制加结尾)
```

1. **`scene_change_times(threshold=0.10)`** — 跑一次上面的命令，解析 stderr 的 `pts_time`，返回有序秒数列表。
2. **`_dedup(min_gap_s=0.8)`** — 时间去重。同一次操作往往引发连续几帧跳变（点击后界面渐变，0.45s、0.74s 都超阈值），其实只算一次操作；把相邻 < 0.8s 的合并取先者，避免对同一动作重复采样。
3. **`_even_sample(max_frames=10)`** — 上限抽样。操作密集时去重后跳变点仍多，超过 10 就在时间轴上近似均匀取 10 个，控制 token 成本。
4. **保底补帧** — 视频很安静时跳变点可能少于 4 个，按视频时长均匀补点（跳过离已有点太近的），保证最低覆盖。
5. **终态保底** — **本方案相对"一般 scene 检测"的关键增量**：无论前面抽了什么，强制把 `duration - 0.1s` 那一帧纳入（离已有帧太近则不重复加）。因为「成没成」几乎只看**最终态**，等间隔常因时长不整除漏掉结尾，纯 scene 也可能因结尾渐变不触发阈值。

这五步是 `select_keyframe_times` 的全部职责，单独函数便于单测（`tests/_verify_op.py` 对真实录屏断言：覆盖发送瞬间、静止段 0 帧、含终态帧、帧数在区间内）。

### 2.4 抽帧执行：`extract_scene_keyframes`（逐时刻精确 seek）

拿到时间点后，**逐个**用精确 seek 抽图，而非一条 `fps` 滤镜：

```
ffmpeg -ss <t> -i video.mp4 -frames:v 1 -vf scale='min(720,iw)':-2 -q:v 3 kf_001.jpg
```

- `-ss <t>`（放 `-i` 前）= 快速 seek 到指定时刻，比 `fps` 滤镜更可控、不漏指定时刻；
- `scale='min(720,iw)':-2` 最长边限 720（竖屏够清晰且省 token）；
- `-q:v 3` 高质量 JPEG。

> 为什么不一条 `fps` 滤镜搞定？`fps` 是固定间隔采样，又回到"等间隔"老路，无法按 scene 选定的时间点抽。所以**先算时间点，再逐点 seek**。

### 2.5 编码：`encode_frame`

`PIL` 等比缩放到最长边 768、JPEG quality 70、`base64` → `data:image/jpeg;base64,...`，直接作 `image_url`。10 帧总体积约 300–350 KB。

### 2.6 参数与调参

| 参数 | 默认 | 含义 | 调整影响 |
|---|---|---|---|
| `threshold` | 0.10 | scene 变化阈值 | 调高→只抽大跳变（更少帧，可能漏小变化）；调低→更敏感（更多帧、更全但更贵） |
| `min_gap_s` | 0.8 | 相邻跳变合并间距 | 调大→合并更多（更稀疏）；调小→保留更多细节 |
| `max_frames` | 10 | 帧数上限 | 超过则均匀抽样；调大更全但 token 更贵 |
| `min_frames` | 4 | 帧数下限 | 不足则补均匀 |
| `max_edge` | 720/768 | 抽帧/编码最长边 | 调小更省 token，但文字可能糊到看不清 |

经验：手机 UI 录屏 `threshold=0.10` 通常正好（界面跳转的 scene 分数多在 0.1–0.4）。若漏抽了"软跳变"（淡入淡出），调低到 0.05–0.08。

### 2.7 对比「一般方案」

| 方案 | 原理 | 操作录屏上的表现 |
|---|---|---|
| **等间隔（fps 滤镜）** | 固定时间间隔均匀采样 | ❌ 静止段重复浪费、跳变段漏帧 |
| **视频 I 帧 / 编码关键帧**（`-skip_frame nokey`） | 取视频压缩编码的关键帧 | ❌ 编码关键帧 ≠ 语义变化：长镜头塞无信息帧、快剪反而漏 |
| **手动帧差分** | 自己算相邻帧像素差 | △ 和 scene 本质相同，但要自实现+调性能；scene 是 ffmpeg 内置优化版 |
| **本方案：scene+去重+上限/保底+终态** | 内容驱动选跳变，再加工 | ✅ 跳变全覆盖、静止 0 帧、帧数可控、终态必抽 |

相对"一般 scene 检测"，本方案还多了三个工程增量：①**去重**（一次操作的连续跳变合并）；②**上限/保底**（scene 数量不可控，用 max/min_frames 钉在可控区间）；③**终态保底**（纯 scene 不保证抽到结尾，而操作评测最看重最终态）。且**纯 ffmpeg + PIL，不依赖 cv2**。

### 2.8 实测对比

同一录屏（闹钟操作，24.79s、720×1590）：

| 策略 | 抽帧时刻（秒） | 静止段(8–14s)帧数 |
|---|---|---|
| 等间隔 8 帧 | 3.08, 6.16, **9.24, 12.32**, 15.40, 18.48, 21.56, 24.64 | 2（浪费） |
| **本方案 10 帧** | 0.45, 4.29, 5.73, 6.78, 14.57, 15.72, 17.12, 20.64, 21.92, 24.69 | **0** |

本方案覆盖了**发送瞬间（0.45s）**（等间隔第一帧要到 3.08s，错过"用户点了什么"），静止段 0 帧（等间隔浪费 2 帧），终态 24.69s 强制纳入。

端到端（官方 `kimi-k2.6`，同一 describe 任务）：关键帧 9 帧 **45s / 3358 tokens** vs 视频直传整段 **73s / 42711 tokens**——快约 38%、省约 12.7 倍，且操作流程/最终态都能准确还原。

### 2.9 视频直传为何不采用

裁判模型 `kimi-for-coding-openai` 经聚合代理（litellm）**只支持 `image_url`，不支持 `video_url`**——传 `data:video/mp4` 被拒（`unsupported image format: video/mp4`；`video_url` part type 也被拒）。Moonshot 官方 API 支持 `video_url` 直传，但贵约 12 倍 token、慢约 60%（见 2.8），性价比远不如抽帧，故本模式走关键帧。

### 2.10 局限

- **阈值需按素材调**：不同帧率/清晰度 scene 绝对值会漂移；必要时可按 scene 分布取分位数自适应。
- **极短跳变仍可能漏**：采样离散，两次选定点之间 < `min_gap_s` 的独立操作可能被合并。
- **VFR（变帧率）seek 精度**：手机录屏常为 VFR，`-ss` seek 偶有偏移。
- **scene ≠ 语义**：scene 只看像素变化，"变化大"不等于"有操作意义"（光照闪烁、加载动画）；可加 OCR 文本变化检测进一步提纯，但成本上升。

## 三、裁判多模态调用

`JudgeClient.complete` 扩展了 `user_images` 参数（`base.py`）：

```python
async def complete(self, system, user, stream_callback=None, user_images=None):
    user_content = user
    if user_images:
        user_content = [{"type": "text", "text": user}] + [
            {"type": "image_url", "image_url": {"url": u}} for u in user_images
        ]
    messages = [{"role": "system", "content": system},
                {"role": "user", "content": user_content}]
```

- 只有**初始 user 消息**变成多模态 content 列表；agent loop 内追加的 assistant/tool 消息仍为字符串，不受影响。
- `_llm_create` / 流式 / 工具循环 / 输出解析**全部复用**——OpenAI SDK 原生支持列表 content，`parse_json_loose` 在文本回复上工作。
- 向后兼容：其他模式不传 `user_images`，行为完全不变。

`RubricJudge.score` 的 operation 分支（`rubric_judge.py`）：从 `EvalItem.metadata["frames"]` 取帧路径，逐个 `encode_frame` 成 data_url，随 `complete(user_images=...)` 喂裁判。

## 四、agent loop 与工具调用

### loop 机制（`base.py` `complete`）

裁判是一个**可多轮调用外部工具的评测智能体**，模仿人类反复查证后再评判：

```
messages = [system, user(含关键帧图)]
for _ in range(max_rounds=12):
    resp = LLM(messages, tools=tool_defs if has_tools)
    if 无 tool_calls: break              # 裁判已确信 → 输出最终 JSON
    执行每个 tool_call → 结果作为 role=tool 消息加回 messages → 继续下一轮
# 若 12 轮仍想调工具（截断）→ 追加一次「无工具」强制判定，保证一定有评分输出
```

### 可用工具（`judges/tools.py` `build_tools`，按裁判配置开关）

| 工具 | 用途 | 默认 |
|---|---|---|
| `web_search` | 联网搜索核查事实/时新信息（tavily/serpapi/bing/jina） | 跟随 judge 配置（judge_1 开） |
| `fetch_page` | 抓取网页正文深挖 | 开 |
| `calculate` | 算术求值核查计算 | 开 |
| `python_run` | 执行代码核查编程/逻辑 | 关 |

### operation 模式实际会调 web_search 吗？

**通常不会**。实测多轮 `used_search=False`。原因：

- **判断操作是否完成靠看图**（关键帧里的界面状态、目标条目是否存在），不依赖外部事实核查；
- `OPERATION_SYSTEM` prompt **没有**像 `RUBRIC_SYSTEM` 那样的「事实核查协议」强制裁判联网核查每个断言——它聚焦于从画面还原 step_trace、比对 evidence。

但**技术上完全支持**：judge_1 的 `enable_web_search=true` 仍生效，agent loop 仍携带搜索工具。如果某条 operation query 涉及可核查的外部事实（如「查询某地日出时间」是否正确），裁判可自主调 `web_search` 核实——这由裁判在 loop 中自主决定，框架不强制。

## 五、时延

以闹钟录屏（24.79s、抽 9–10 帧、单裁判）实测：

| 阶段 | 典型耗时 | 说明 |
|---|---|---|
| 上传 + 抽帧（`/api/upload/video`） | ~2–3 s | 含 ffmpeg 场景检测 + 逐帧 seek 抽图 |
| 帧编码（PIL base64 ×10） | ~1 s | |
| 裁判单次 `complete`（吃多图） | **45–66 s** | 推理模型，波动较大（官方 kimi-k2.6 约 45s，代理 kimi-for-coding-openai 约 66s） |
| 单题总计（1 裁判） | **~50–70 s** | |

**video 直传 vs 关键帧**（官方 `kimi-k2.6`，同一视频、同一 describe 任务公平对比）：

| 方案 | 耗时 | prompt tokens | 效果 |
|---|---|---|---|
| video_url 直传 | 73 s | 42,711 | 带精确时间轴、捕捉帧间转瞬动作 |
| 关键帧 image_url（9帧） | **45 s** | **3,358** | 准确还原操作流程 + 最终态，无时间轴 |

→ 关键帧**快约 38%、省约 12.7 倍 token**，效果对"操作是否完成"判定已足够；video 直传仅极少数需时间轴的高价值单条才值得。注意：代理 `kimi-for-coding-openai` **不支持** video_url（litellm 报 `invalid part type`），video 直传只能走官方 API。

> 注意：`kimi-for-coding-openai` 是**推理模型**，响应先输出 `reasoning_content`（思考）再输出 `content`（答案）。`max_tokens` 给小了（如 800）会全花在 reasoning 上、`content` 为空——`complete` 默认给足上限。
>
> 该模型还被代理限定 `temperature=1`（其他值直接 400），故输出有随机性——框架的「重复采样 + Bootstrap CI + 一致率标红」机制正是为此抗噪。

`/api/eval` 是异步任务（`asyncio.create_task`），立即返回 `task_id`，前端用 SSE（`/api/eval/{task_id}/stream`）实时接收每题结果，不会阻塞。

## 六、评分维度配置

维度**完全配置驱动**，来自 `config/skills/operation.yaml`（自动被 `_load_skills` 加载到 `domain_skills["operation"]`）：

```yaml
name: operation
display: 操作类
rubrics:
  - name: 操作完成度
    description: 是否真正执行并完成意图；区分「真的做了」与「只给口头指导/教程而未执行」
    weight: 1.5
    scale: 5
  - name: 步骤正确性
    weight: 1.0
  - name: 最终态正确
    weight: 1.2
  - name: 效率与稳健性
    weight: 1.0
```

- `OPERATION_SYSTEM` 用 `{% for d in dims %}` **动态渲染**维度名与 JSON 输出格式，维度来自 yaml，不再硬编码。
- **加减/改维度只需改 yaml**（重启 server 生效），无需动代码。
- `weight` 进入 `aggregate_scores` 的加权总分；`scale` 进入 low_agreement 阈值计算。
- 输出格式为「一级维度直接数值」（`{"操作完成度": 4}`），因为 `_flatten_rubric` 对无子维度的一级维度只认直接数值——per-dim 理由并入 `rationale`。

## 七、数据结构与接口

### JSONL 批量导入

操作类页面支持导入 JSONL，每行一条操作任务：

```jsonl
{"id":"op_alarm_001","query":"设置一个明天早上7点的闹钟","context":"当前时间为22:00，设备时区为北京时间","video_path":"data/operation_videos/alarm_001.mp4","agent_statement":"已设置完成"}
{"id":"op_wifi_002","query":"关闭手机无线局域网","video_path":"data/operation_videos/close_wifi.mp4"}
```

| 字段 | 必填 | 说明 |
|---|---:|---|
| `id` | 否 | 缺失时提交评估前自动生成；同一清单中不可重复 |
| `query` / `question` | 是 | 操作任务 |
| `context` | 否 | 时间、地点、设备状态等可信背景 |
| `video_path` | 是 | 本地视频路径；相对路径以项目根目录为基准 |
| `agent_statement` | 否 | agent 自述；解析后映射到内部 `answer` 字段 |

用户不需要提供 `frames`。导入 JSONL 时只解析和校验字段；点击“开始评估”后，每道题才会在
自身并发槽内完成路径校验、`ffprobe` 检查和关键帧抽取。某条视频失败时只标记该题失败，
同批其他题继续评估。

每次 Web 评估使用同一个可排序会话名：

```text
runs/web_history/20260717_103930_operation_aa5cd32001ec.json
runs/videos/imported/20260717_103930_operation_aa5cd32001ec/001_slow_query_001/kf_001.jpg
```

旧的 `runs/web_history/<task_id>.json` 仍可加载、导出和删除，不会自动改名。重复启动同一任务内部
重试时会复用该会话题目目录中带 `.complete` 标记的完整帧；新评估会创建自己的会话目录。

批量清单默认只能读取项目根目录内的文件，防止任意本地文件读取。需要引用外部目录时，
在 `.env` 的 `OPERATION_VIDEO_ROOTS` 中显式加入允许目录；macOS/Linux 多目录以 `:` 分隔，
Windows 以 `;` 分隔。

- `EvalItem`（`schema.py`）新增 `media: list[str]`（视频/图片路径）；抽帧结果缓存到 `metadata["frames"]`（帧路径列表）。
- `web/runner.py` `_to_evalitem` 把 `media` 和 `frames` 透传进 `EvalItem`。
- `POST /api/upload/video`（`server.py`）：`UploadFile`（限 ≤20MB）→ 存 `runs/videos/<uuid>.mp4` → 场景抽帧到 `runs/videos/<uuid>_frames/` → 返回 `{video_id, video_path, frames[], frame_count, duration}`。
- `POST /api/operation/prepare`：保留的兼容接口；新 Web 批量导入流程不再调用它。
- 帧留存与回溯：手动上传帧保存在 `runs/videos/<uuid>_frames/`；批量帧按上述 Web 会话目录保存。裁判 trace（`runs/judge_calls.jsonl`）会等 Web 单题结果生成后再落盘，沿用单回答盲评的结果格式：操作意图、完成判定、总分、步骤与证据、耗时等结果字段在顶层；动态评分维度保留在 `rubric`，维度理由保留在 `rubric_reasons`；同时保留完整 `llm_rounds` 并新增 `model_raw_output`。`image_url` base64 仍脱敏为 `[frame → 路径]`，避免日志膨胀。
- 前端（`static/app.js` + `index.html`）「操作类(录屏)」tab：逐题上传仍在上传时抽帧；JSONL 批量导入只解析，开始评估时抽帧。

## 八、设计取舍与已知限制

- **抽帧 vs 视频直传**：走抽帧（代理支持 image_url），便宜（10 帧 ≈ 3k token）但可能漏掉两次采样间的极短跳变；场景检测 + 终态保底已显著缓解。视频直传（官方 API）保真但贵约 14 倍，架构已预留（`complete` 的 list content 可扩展 `video_url` block），按需启用。
- **agent 自述交叉验证**：可填 agent 自述（如「我已设好闹钟」），裁判做「自述 vs 画面证据」比对，抓「只说不做」的幻觉。
- **仲裁不带帧**：多裁判分歧触发主席仲裁时，仲裁走纯文本（不看帧），属兜底；单裁判或一致时不触发。
- **裁判视觉能力**：operation 默认只用 `judge_1`（kimi-for-coding-openai，已验证支持 image_url）。其他裁判（如 deepseek）是否支持图片需接入时验证，不支持则前端不勾选。
- **temperature 随机性**：kimi 强制 `temperature=1`，建议用多裁判 + 重复采样抗噪。
