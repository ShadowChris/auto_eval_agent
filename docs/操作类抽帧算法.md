# 操作类抽帧算法

本文档是操作类录屏抽帧算法的版本基线和迭代记录。评测整体流程、裁判提示词和
Web 使用方式仍参考 `docs/操作类评测.md`；本文只描述
`src/auto_eval/media.py` 中的关键帧算法。

| 项目 | 值 |
|---|---|

版本号同时写入每次抽帧生成的 `keyframes.json`，并参与 Web 抽帧缓存键计算。
修改算法版本或任务起止时间都会使旧缓存失效，避免复用不同策略生成的帧。

后续版本约定：

- 在现有算法上补充候选状态或调整兼容参数，升级次版本，例如
  `hybrid-state-v3.1.0`。
- 对状态建模或抽帧流程进行不兼容重构，升级主版本，例如
  `hybrid-state-v4.0.0`。

## 2. 输入时间语义

JSONL、Web 传递参数、抽帧函数和配置类统一使用以下字段，单位均为秒：

| 字段 | 默认值 | 含义 |
|---|---:|---|
| `task_start_time` | `7.0` | query 大致发送完成、任务开始执行的时间 |
| `task_end_time` | `null` | 显式任务结束点；为空时由算法自动推断 |

两个字段均可省略或填 `null`。显式值必须是有限的非负数字，且
`task_end_time` 必须大于 `task_start_time`，同时不能超出视频时长。
即使填写了任务结束点，算法仍会强制保留视频最终帧，用于确认任务结束后返回的界面状态。

## 3. 当前算法流程

### 3.1 生成候选帧

在任务时间窗内合并以下来源：

1. `task_start_time` 对应的任务首帧；
2. `sample_fps=1.0` 的固定频率采样帧；
3. FFmpeg `scene` 检测产生的画面变化帧；
4. 显式或自动推断的任务结束帧；
5. 距视频末尾约 0.3 秒开始尝试解码的最终帧。

固定采样负责避免只依赖视频编码或场景阈值导致的漏检，`scene` 候选负责补充两次
固定采样之间的快速变化。

### 3.2 合并相邻时间点

时间差不超过 `timestamp_merge_gap_s=0.25` 的候选视为同一事件，优先保留任务首帧、
任务结束帧和 scene 帧，减少同一次界面跳变产生的近重复帧。

### 3.3 UI 状态聚类

每张候选帧会经过高斯模糊并缩小为 `16×27` 的灰度布局特征。相邻帧的布局距离达到
`state_layout_threshold=0.05` 时划分为新的 UI 状态。

对于持续时间达到 `stable_min_duration_s=1.5` 的状态，保留能够表示状态开始或结束的
候选帧。该步骤用于合并同一界面中的光标、计时器、轻微文字变化和录屏计时变化。

### 3.4 短暂弹窗保护

对 A → B → A 型局部变化进行额外保护。当 B 持续时间不超过
`transient_max_duration_s=4.0`，且变化面积、返回相似度满足阈值时，B 会标记为
`transient-local-A-B-A`，避免静音、24 小时制等短暂卡片被状态去重吞掉。

### 3.5 自动任务结束点

未填写 `task_end_time` 时，算法比较候选状态与最终智能助手外壳的相似度，结合最终状态
稳定时长和置信度推断任务结束点。推断成功的帧标记为 `task-end-auto`；置信度不足时
不强行截断任务区间。

### 3.6 帧数控制和最终严格去重

候选帧最多保留 `max_frames=16`。最终去重使用清晰图像特征，而不是模糊布局特征：

- RMS 阈值：`final_dedup_rms_threshold=0.008`
- 变化像素占比阈值：`final_dedup_changed_fraction_threshold=0.01`

任务结束帧和视频最终帧属于保护帧。视频最终帧始终保留，即使它与前一帧相似。

## 4. 主要默认参数

| 参数 | 默认值 | 调大后的主要效果 |
|---|---:|---|
| `max_frames` | 16 | 允许保留更多状态，但增加视觉输入成本 |
| `sample_fps` | 1.0 | 更容易捕获短变化，同时增加候选计算量 |
| `scene_threshold` | 0.06 | scene 候选更少，更容易漏掉小变化 |
| `scene_min_gap_s` | 0.8 | 合并更多相邻 scene 事件 |
| `timestamp_merge_gap_s` | 0.25 | 合并更多时间接近的候选 |
| `state_layout_threshold` | 0.05 | UI 状态更难被划分，输出帧更少 |
| `stable_min_duration_s` | 1.5 | 更少保留短状态 |
| `transient_max_duration_s` | 4.0 | 允许保护持续更久的 A-B-A 弹窗 |
| `auto_task_end_confidence_threshold` | 0.60 | 自动任务结束判断更保守 |
| `final_dedup_rms_threshold` | 0.008 | 最终去重更激进 |
| `final_dedup_changed_fraction_threshold` | 0.01 | 最终去重更激进 |

完整配置及其默认值以 `KeyframeConfig` 为准。调整参数时必须重跑 mini 数据集，不应只针对
单个案例调参。

## 6. 已知待优化点

### P0：持续型顶部卡片可能被同界面聚类吞掉

当前 mini 数据集中有 3 条明确案例：

| 案例 | 当前结果 | 漏掉的信息 |
|---|---|---|
| `fast_query_30` | 7.0、10.0、96.125 秒，共 3 帧 | 约 18–82 秒持续显示的顶部操作卡片 |
| `fast_query_34` | 7.0、51.561 秒，共 2 帧 | 约 14–39 秒持续显示的字体设置卡片 |
| `fast_query_95` | 7.0、9.0、56.446 秒，共 3 帧 | 约 14–44 秒持续显示的卡 2 操作卡片 |

根因：

1. 卡片主要改变助手界面的局部区域；
2. 布局特征经过高斯模糊并缩小为 `16×27` 后，对局部卡片不够敏感；
3. `fast_query_34` 的最大相邻布局距离约为 `0.0475`，低于当前 `0.05` 阈值，整段被
   聚成一个状态；
4. 这些卡片持续时间大于 4 秒，不符合当前短暂 A-B-A 弹窗保护条件；
5. 它们在状态聚类阶段就没有进入最终候选，因此问题不在最终严格去重。

### 推荐优化方向

增加“持续型局部状态”检测，而不是直接降低全局 `state_layout_threshold`：

1. 保留现有低分辨率布局特征，用于区分整页或应用级变化；
2. 增加清晰图像的局部变化特征，重点观察顶部卡片、底部弹窗等区域；
3. 当局部变化连续稳定至少 1.5 秒时，将其作为新的持久状态候选；
4. 对文字逐行生成、滚动和录屏计时器变化继续去重；
5. 新候选仍进入 `max_frames` 限制和最终严格去重。

不建议直接把全局布局阈值从 `0.05` 下调，因为这会同时放大普通文字变化、滚动和加载动画，
可能重新增加大量重复帧。

### 下一版本验收条件

计划版本：`hybrid-state-v3.1.0`。

1. `fast_query_30/34/95` 各至少保留一张包含关键顶部卡片的帧；
2. `fast_query_4/5/7/28` 当前已保留的关键状态不能回退；
3. 17 条全部保留任务首帧和视频最终帧；
4. 单案例不超过 16 帧；
5. mini 总帧数建议不超过 86 帧，即相对 v3.0.0 基线增长不超过约 20%；
6. 新增或调整去重规则时，必须记录逐案例帧数和时间点差异。

## 7. 测试与回归入口

- `tests/test_media.py`：算法配置、严格去重和真实 FFmpeg 视频集成测试；
- `tests/test_operation_import.py`：JSONL 时间字段、参数透传、缓存失效和视频时长校验；
- `data/test_operation_fast_query_mini.jsonl`：快速人工回归数据集；

版本升级时，应先更新实现和测试，再完整重跑 mini 数据集，最后把结果追加到本文档，保留
旧版本基线，不覆盖历史数据。
